## 一、面试全景：2026 年企业到底在考什么

### 1.1 考察重心的三次迁移

| 阶段 | 面试关键词 | 淘汰线 |
|------|-----------|--------|
| 2023「会调 API」 | Prompt、OpenAI SDK、简单 Demo | 能跑通就行 |
| 2024-2025「会搭系统」 | RAG 链路、Agent 框架、向量库、微调 | 讲得出链路与选型 |
| 2026「会治理落地」 | 检索质量归因、Agent 可控性、评测闭环、成本归因、MCP 治理、安全护栏 | 讲得出**排查路径、量化数据、Trade-off** |

2026 年面试官的典型心态：**默认你能搭出 Demo，重点验证你能不能把 Demo 变成可运营的生产系统**。所以「是什么」类问题占比下降，「出了问题怎么排查」「为什么这么选而不是那样选」「上线后效果变差怎么定位」类问题占比大幅上升。

### 1.2 三层考察模型

1. **第一层：大模型基础认知**——Token / 上下文 / 采样参数 / 幻觉 / 结构化输出。考察你是否理解这些参数如何影响**成本、延迟、稳定性和线上质量**，不是背定义。
2. **第二层：AI 应用组件能力**——RAG、Agent、Prompt、MCP、工具调用、记忆。考察你**落地过没有、出了问题会不会排查、为什么这么选**。
3. **第三层：AI 系统设计**——知识库问答系统、Agent 平台、模型网关、评测体系。考察**架构分层、治理策略、演进路线**，社招 2 年以上几乎必问。

### 1.3 不同经验层级的深度要求

| 层级 | 必须达到的深度 | 面试官验证方式 |
|------|--------------|---------------|
| 应届 / 0-1 年 | 讲清基本链路：文档 → 切片 → Embedding → 检索 → 生成；Agent 与 Chatbot / Workflow 的区别；知道 Demo 与生产之间有工程差距 | 让你画链路图、解释项目里每个组件的作用 |
| 1-3 年 | 能定位问题、能做取舍：RAG 按段排查、工具调用失败降级、结构化输出失败处理、Prompt 新旧版本如何证明谁好 | 场景题：「召回率低怎么查」「知识库更新后旧答案还在怎么办」 |
| 3 年以上 | 完整架构 + 治理 + 质量闭环：模型路由 / fallback / 成本归因 / 权限隔离 / 灰度回放 / 评测门禁 | 开放系统设计题 + 连环追问「流量翻 100 倍怎么办」「供应商限流怎么降级」 |

### 1.4 通用答题框架

```
1. 概念：一句话讲清它是什么（证明你懂）
2. 问题：它在真实系统里会引起什么（证明你用过）
3. 方案：生产环境怎么设计 / 排查 / 优化 / 治理（证明你做过）
4. 边界：什么场景适用、什么场景不适用（证明你有判断力）
```

以「RAG 召回率低怎么优化」为例：

- ❌ 差回答：直接背「上 Hybrid Search、加 Rerank、做 Query 改写」——组件名堆砌，面试官无法确认你做过。
- ✅ 好回答：「先判断正确证据有没有进候选池：没进池，排查文档解析 → 切片 → Embedding → 元数据过滤 → Query 改写；进了池但排得靠后，再上混合检索、Rerank、调候选池大小和融合权重；证据进了上下文答案还错，才查 Prompt、上下文位置和模型忠实度，用评测集固化。」——按**归因顺序**答，一听就是做过系统的人。

### 1.5 面试官的扣分雷区（通用）

1. 只报组件名不讲取舍：「我们用了 LangChain + Milvus + bge」——追问「为什么不用 pgvector」立刻卡壳。
2. 没有任何量化数据：讲优化不带前后对比（召回率、延迟、成本、命中率）。
3. 把 Demo 说成生产：追问 QPS、并发、灰度、回滚、告警时露馅。
4. 归因错位：答「召回不准就改 Prompt」这类跨段乱治，直接暴露没做过归因。
5. 声称跑过没跑过的评测：追问指标定义（如 Faithfulness 怎么算的）时答不出。

## 二、大模型基础（所有 AI 岗必问）

> 考察定位：过滤「只会复制 Demo」的候选人。每一题都在验证你是否理解参数与工程指标（成本 / 延迟 / 稳定性）之间的因果关系。

### 2.1 Token 与上下文窗口

**考点**：Tokenizer 原理（BPE）、中英文 Token 差异、上下文窗口限制与成本、长上下文有效性。

**高频问法**
- 「Token 是什么？为什么同样一段话，中文比英文费 Token？」
- 「上下文窗口 128K 是什么意思？是不是越大越好？」
- 「你们项目里怎么估算和控制 Token 消耗的？」
- 「模型宣称支持 1M 上下文，实际能用吗？有什么坑？」

**追问链**：Token 怎么计费 → 输入输出价格为什么不一样 → 上下文塞满会发生什么 → Lost in the Middle 是什么 → 工程上怎么缓解（重排关键信息位置 / 压缩 / 分段）。

**答题要点**
- BPE 子词切分；中文常见 1 字 ≈ 0.6~1 Token，英文 1 词 ≈ 1.3 Token；代码因缩进符号更费。
- 上下文越长：成本线性涨、首 Token 延迟（TTFT）涨、中段信息召回率下降（Lost in the Middle，RULER 基准显示多事实并列时召回明显衰减）。
- 生产控制手段：历史消息裁剪 / 滚动摘要、RAG 只注入 top-k、max_tokens 限制输出、Prompt 前缀缓存。
- 能报出自己项目的估算方法（如按字符系数估算、tiktoken 精确计数）与预算兜底策略是加分项。

**扣分点**：答「窗口越大越好」；不知道输入与输出 Token 价格差异；讲不出任何控制手段。

### 2.2 采样参数（Temperature / Top-P / Top-K / 惩罚项）

**考点**：每个参数控制什么、生产环境怎么配、与业务场景的对应关系。

**高频问法**
- 「Temperature 和 Top-P 分别控制什么？两个一起调会怎样？」
- 「代码生成、客服问答、创意文案分别怎么配采样参数？」
- 「为什么线上要把 Temperature 调低？调成 0 就完全确定了吗？」
- 「frequency_penalty 和 presence_penalty 有什么区别？」

**答题要点**
- Temperature 缩放 logits 分布（越低越保守）；Top-P 按累积概率截断候选集；Top-K 按数量截断。一般只调其一，避免联合调参不可解释。
- 场景经验值：结构化抽取 / 分类 / 工具调用 T≈0~0.2；知识问答 T≈0.3~0.7；创意生成 T≈0.8+。
- T=0 也不完全确定：浮点并行归约、MoE 路由、服务端 batch 组合都会引入非确定性——这是 2026 常见加分追问。
- 惩罚项：frequency 按出现次数累计惩罚（防复读），presence 出现过即惩罚（促新话题）。

### 2.3 幻觉（Hallucination）

**考点**：幻觉根因分类、工程缓解手段、与 RAG / 评测的联动。

**高频问法**
- 「大模型为什么会一本正经地胡说八道？」
- 「你们项目里怎么减少幻觉的？效果怎么量化？」
- 「RAG 之后还有幻觉吗？怎么处理？」
- 「用户拿着一条 AI 编造的答案投诉，你怎么排查？」

**追问链**：幻觉根因 → 检索增强后为什么还编 → Faithfulness 怎么度量 → 引用溯源怎么做 → 兜底话术与置信度策略。

**答题要点**
- 根因：训练目标是「预测下一个 Token」而非「陈述事实」；知识截止；采样随机性；指令诱导（用户要求它必须给答案）。
- 工程缓解四板斧：RAG 提供证据、Prompt 约束（「仅基于给定上下文回答，不知道就说不知道」）、结构化引用溯源（答案句关联 chunk id）、离线评测（Faithfulness）+ 线上抽检。
- RAG 后仍会幻觉：证据不足时模型补脑、跨 chunk 拼接错误、忽略上下文用参数记忆——所以要评 Faithfulness 而不是只评召回。

### 2.4 结构化输出三件套：JSON Mode / Structured Outputs / Function Calling

**考点**：三者边界、失败兜底、与业务解析层的配合。（2026 高频，送分题答不好变送命题）

**高频问法**
- 「JSON Mode 和 Structured Outputs 有什么区别？」
- 「Function Calling 的完整调用链路是什么？模型真的执行函数吗？」
- 「模型返回的 JSON 解析失败怎么办？你们线上怎么兜底的？」
- 「什么时候用 Prompt 约束输出格式，什么时候用 Function Calling？」

**答题要点**
- JSON Mode 只保证「合法 JSON」，不保证字段符合 Schema；Structured Outputs 用约束解码保证符合 JSON Schema；Function Calling 是模型输出「调用意图 + 参数」，**执行永远在你的代码里**。
- 生产兜底链：约束解码 / 提供 Schema → 解析失败先做 JSON 修复（去 markdown 围栏、补引号、json-repair 库）→ 重试并把错误信息回填 Prompt → 多次失败降级默认值 + 记日志入 bad case。
- Pydantic 校验放在解析之后：格式合法 ≠ 业务合法（枚举值、范围、互斥字段）。

**扣分点**：说「Function Calling 就是模型帮你执行函数」；线上没有任何解析失败兜底。

### 2.5 流式输出：SSE 与 WebSocket

**考点**：协议选型、SSE 事件设计、断连与终态处理。（前后端联动高频考点）

**高频问法**
- 「为什么大模型产品都用 SSE 而不是 WebSocket？」
- 「SSE 的报文格式是什么样的？event 和 data 字段怎么用？」
- 「用户网络断了，后端的 LLM 调用还在继续吗？要不要取消？」
- 「SSE 经过 Nginx 后变成一次性输出，怎么排查？」

**追问链**：SSE 单向 vs WS 双向 → 事件类型设计（delta / reference / card / error / done）→ 断线重连（Last-Event-ID）→ 服务端感知断连（`request.is_disconnected`）→ 已受理任务是否随断连取消（工作流类任务不应取消，聊天类可取消）→ Nginx `proxy_buffering off` / `X-Accel-Buffering: no`。

**答题要点**
- LLM 输出是典型「服务端单向推送」，SSE 基于 HTTP、天然过网关、自动重连、实现简单；需要双向交互（语音、协作编辑）才用 WebSocket。
- 事件流必须有**显式终态**（done / error 事件），前端不能靠连接关闭猜测结束；错误也要作为事件推送而非直接断流。
- 断连语义要分场景：对话流可取消节省成本；已受理的长任务（工作流）断连不取消，用户重连后按 run id 恢复。

### 2.6 LLM API 工程化调用

**考点**：超时、重试、幂等、限流、降级——把 LLM 当成一个不稳定的第三方服务来治理。

**高频问法**
- 「调用 LLM API 超时时间设多少？重试策略怎么定？」
- 「哪些错误可以重试，哪些不能？」
- 「主模型挂了怎么办？你们的降级链路是什么？」
- 「多个用户并发打满了供应商限流（429）怎么处理？」

**答题要点**
- 只重试可恢复错误（429 / 5xx / 超时 / 连接错误），指数退避 + 最大次数 + 重试日志；400 / 401 / 403 重试无意义。
- 流式调用的超时要拆：连接超时、首 Token 超时（TTFT）、Token 间隔超时，整体兜底超时。
- 降级链：主模型（如 deepseek-v4-flash）→ 备选国产模型（qwen-plus / glm）→ 缓存答案 / 模板兜底；降级要有开关与观测。
- 幂等：写操作（扣预算、落库）用请求 id 去重，避免重试造成重复扣费。

### 2.7 DeepSeek 与国产模型专题（2026 新热点）

**考点**：国内企业技术栈合规（信创 / 数据出境）背景下，国产模型 API 的工程特性。

**高频问法**
- 「deepseek-chat 和 deepseek-reasoner（思考模式）有什么区别？什么任务用哪个？」
- 「DeepSeek 的上下文硬盘缓存（Context Caching）是什么？怎么设计 Prompt 才能提高缓存命中？」
- 「为什么 DeepSeek 便宜这么多？知道 MLA / MoE 吗？」（进阶，见第八章）
- 「国产模型和 GPT 系在 Function Calling / 结构化输出上的差距，你实际遇到过什么坑？」
- 「你们为什么选 DeepSeek 而不是通义 / 智谱？选型维度是什么？」

**答题要点**
- 模型现状（2026-07）：官方主推 deepseek-v4-flash / v4-pro（1M 上下文，支持思考 / 非思考模式切换）；旧模型名 deepseek-chat / deepseek-reasoner 已成为 v4-flash 非思考 / 思考模式的兼容别名，2026-07-24 起弃用。任务分配逻辑不变：抽取 / 客服 / 常规 RAG 用非思考模式（快而便宜），数学 / 复杂规划 / 代码难题开思考模式（思维链 Token 也按输出计费、更慢），思考强度用 `reasoning_effort` 参数控制（high / max，官方建议复杂 Agent 场景开思考模式并设 max）；思考模式下部分采样参数不生效。面试报模型名时体现出「知道弃用与别名关系、V4 用参数而非模型名切换思考模式」是时效性加分。
- Context Caching：命中条件是**从第 0 个 token 起前缀完全相同**（按块粒度对齐），命中部分按缓存价计费——当前命中价约为未命中价的 1/50（v4-flash）~ 1/120（v4-pro），即便宜约两个数量级（以官网现价为准），是成本优化第一杠杆。工程做法：**System Prompt、Few-shot、工具定义放前缀且保持稳定，动态内容（用户输入、检索结果）放后面**；多轮对话天然前缀复用；通过 usage 里的 `prompt_cache_hit_tokens` 监控命中率。
- 峰谷分时定价（2026-07 新机制，以官网为准）：官方已向开发者预告随 V4 正式版启用分时计价——北京时间 9:00–12:00 / 14:00–18:00 峰时 API 价格翻倍，其余时段维持基准价。成本答法新增一条杠杆：**离线批量任务（文档入库、评测夜跑、批量抽取）错峰调度吃谷时价**；命中 / 未命中价同倍缩放，比例不变，缓存优化与错峰可叠加。
- FC / 结构化输出的实际差距（对应上面「踩过什么坑」问法）：DeepSeek 支持 JSON Output 与 Tool Calls，但 JSON mode 要求 **Prompt 中必须出现 "json" 字样并给出输出格式示例**，且只保证合法 JSON——没有 OpenAI 式 `json_schema` 约束解码，字段级契约要靠 Pydantic 校验 + 修复重试兜底；官方文档明示 JSON mode 有概率返回空 content（需检测重试）。答法落点：**不赌单次输出，靠「解析 → 校验 → 错误回填重试 → 降级」链兜底**（呼应 2.4），并在自己场景实测 FC 稳定性后再定方案。
- 选型维度：任务效果实测（自建评测集）、价格、延迟、Function Calling 稳定性、并发配额、合规要求——答「跑过自己场景的评测集」远好于「看榜单」。

## 三、Prompt Engineering 与 Context Engineering

> 考察定位：Prompt 是最容易吹水的领域，所以面试官会用「工程化管理」和「效果证明」来验真：你的 Prompt 是文件还是代码里的字符串？改了 Prompt 怎么证明变好了？

### 3.1 Prompt 设计原则与常用模式

**考点**：结构化 Prompt、角色设定、分隔符、输出约束、少样本设计。

**高频问法**
- 「说说你写 Prompt 的方法论，别只说多试。」
- 「一个生产级的 System Prompt 应该包含哪些部分？」
- 「Few-shot 示例怎么挑？例子越多越好吗？」
- 「Zero-shot 和 Few-shot 什么时候用哪个？」

**答题要点**
- 生产 System Prompt 骨架：角色与目标 → 能力与工具边界 → 工作步骤 → 输出格式（Schema / 示例）→ 约束与禁止项 → 兜底行为（不知道怎么办）。
- 分隔符隔离（XML 标签 / 三引号）用户输入与指令，防注入也防歧义。
- Few-shot：示例要覆盖边界情况（含「拒答」示例）、与真实输入同分布、顺序影响输出；示例多了费 Token 且可能过拟合格式，一般 2~5 个。
- 关键原则：告诉模型「做什么」优于「别做什么」；先给证据再提问；让模型「想清楚再回答」（CoT 触发）。

### 3.2 CoT / ReAct / 推理模式

**考点**：CoT 原理与适用性、Self-Consistency、ReAct 与推理模型时代的变化。

**高频问法**
- 「Chain-of-Thought 为什么有效？所有任务都要加吗？」
- 「ReAct 的 Prompt 长什么样？Thought / Action / Observation 怎么组织？」
- 「有了 DeepSeek-R1 这类推理模型，还需要手写 CoT 吗？」
- 「Self-Consistency 是什么？什么时候值得花这个成本？」

**答题要点**
- CoT 把隐式推理显式化成中间 Token，给模型「计算空间」；简单任务加 CoT 浪费 Token 还可能降准确率。
- 推理模型时代的新答法（2026 加分）：R1 / o 系列已内化 CoT，手写「一步步思考」收益下降甚至干扰；应用层重心转向**给足上下文与约束**，把「怎么想」交给模型。
- Self-Consistency：多次采样取多数，适合有唯一正确答案且错误成本高的任务（数学 / 抽取），成本乘 N 倍。

### 3.3 Prompt 注入与防御

见 [11.2 Prompt 注入](#十一安全合规与护栏)，此处只列 Prompt 层高频问法：

- 「用户输入『忽略以上所有指令』怎么办？」
- 「RAG 检索回来的文档里藏了恶意指令（间接注入），你怎么防？」
- 「System Prompt 被套出来了有什么风险？怎么降低泄露面？」

### 3.4 Prompt 工程化管理（版本 / A/B / 评测）

**考点**：Prompt 即资产：文件化存储、版本控制、变更验证、灰度上线。

**高频问法**
- 「你们的 Prompt 存在哪？改一次 Prompt 的完整流程是什么？」
- 「怎么证明新版 Prompt 比旧版好？拍脑袋还是有数据？」
- 「线上跑着的 Prompt 出了 bad case，你的处理闭环是什么？」
- 「Prompt 模板里的变量注入怎么做的？为什么不用 f-string 拼接？」

**答题要点**
- 存储：模板文件（.txt / .jinja2 / .yaml）+ Git 版本控制，禁止硬编码在业务代码；模板引擎注入变量并对用户输入消毒。
- 变更验证：离线评测集（golden set）跑分对比 → 通过后灰度 A/B（按用户桶）→ 观察业务指标与护栏指标 → 全量；重大变更保留一键回滚。
- bad case 闭环：线上采集 → 归因（Prompt / 检索 / 模型）→ 修复 → **回灌评测集防止复发**。
- 能讲出「Prompt A/B 并排打分 + 版本管理」的实际做法（如自建 Prompt 调试台）是强加分。

### 3.5 Context Engineering（2026 新热点）

**考点**：从「写好一句话」到「管理整个上下文窗口」的工程化转变——2026 年 JD 高频词。

**高频问法**
- 「Prompt Engineering 和 Context Engineering 有什么区别？」
- 「一次 Agent 调用的上下文里都有什么？怎么排布优先级？」
- 「上下文快满了怎么办？压缩策略有哪些，各有什么损失？」
- 「多轮对话 / 长任务里，你怎么决定什么进上下文、什么进外部存储？」

**答题要点**
- 定义区分：Prompt Engineering 优化「指令怎么写」；Context Engineering 优化「窗口里放什么、放哪、放多少」——涵盖系统指令、工具定义、记忆、检索结果、历史消息的动态编排。
- 上下文分层：常驻层（System + 工具 Schema，尽量稳定以吃前缀缓存）→ 任务层（当前目标、计划、关键状态）→ 动态层（检索片段、最近对话）→ 溢出层（外部文件 / 数据库，按需检索回来）。
- 压缩策略与损失：滚动摘要（丢细节）、截断旧轮次（丢远期依赖）、去重与折叠工具输出（低损失，优先做）、把大结果写外部存储只留引用（Agent 场景标配）。
- 经验红线：上下文利用率长期贴着上限跑，质量会先于报错崩掉（注意力稀释），预留余量并触发主动压缩。

## 四、RAG 全链路

> 考察定位：RAG 是企业 AI 面试的第一主战场，几乎每家必问。面试官的核心验证点是**分段归因能力**：入库（解析/切片/向量化）、检索（召回/融合/精排）、生成（Prompt/忠实度）三段你能不能独立排查、独立评测。

### 4.1 RAG 基础与整体链路

**高频问法**
- 「画一下你们 RAG 系统的完整链路，从文档上传到答案返回。」
- 「RAG 和微调怎么选？什么时候两个都要？」
- 「长上下文模型（1M window）出来了，RAG 还有必要吗？」
- 「RAG 每一段的瓶颈分别在哪？」

**答题要点**
- 三段式链路：**入库**（Loader → 切片 → Embedding → 向量库 + 元数据）→ **检索**（Query 改写 → 多路召回 → 融合 → Rerank → top-k）→ **生成**（上下文注入 → LLM → 引用溯源 / 后处理）。
- RAG vs 微调：知识频繁更新 / 需要溯源 / 数据量小 → RAG；改变行为风格、领域语言习惯、格式规范 → 微调；企业常见组合是「微调调风格 + RAG 供事实」。
- 长上下文 vs RAG：互补不替代。单文档深度理解走长上下文；跨文档、增量更新、高 QPS、需权限过滤走 RAG；折中方案是 RAG 粗筛 + 长上下文精读。长上下文的坑：贵、慢、Lost in the Middle、无法做权限隔离。
- 各段瓶颈：切片破坏语义 / Embedding 领域词失效 / 召回只看相似不看相关 / Rerank 成本 / 上下文顺序影响 / 生成不忠实。

### 4.2 文档解析与切片（Chunking）

**高频问法**
- 「PDF 里的表格、双栏排版怎么处理？解析错了对下游影响多大？」
- 「chunk_size 和 overlap 怎么定的？为什么是 500 不是 1000？」
- 「固定切片、递归切片、语义切片的区别和适用场景？」
- 「切片把一句话切断了怎么办？标题和正文分家了怎么办？」

**追问链**：切片策略 → 中文切片特殊性（标点优先、避免词中断开）→ 父子切片（小块检索大块生成）→ Contextual Retrieval（给 chunk 加上下文前缀）→ 你们怎么验证切片质量（黄金集上对比 Recall）。

**答题要点**
- 默认选择：递归 / 结构化切片（按标题、段落、代码块边界），保留元数据（标题路径、页码、来源）；固定切片只做 baseline。
- 语义切片按句间 Embedding 相似度断开，成本高，长文专题才用；Late Chunking / Contextual Retrieval 是 2026 加分项（Anthropic 实测 top-20 检索失败率：上下文化 Embedding 降 35%（5.7%→3.7%），叠加上下文化 BM25 共降 49%（→2.9%），再加 Rerank 共降 67%（→1.9%））。
- 关键参数经验：中文常见 300~800 字符 + 10~20% overlap，但正确答法是「**在自己的评测集上扫参对比**，不是抄经验值」。
- 表格 / 扫描件：版式解析（如 MinerU / OCR）→ 表格转 Markdown 或按行拆 + 表头继承；解析错误是很多「检索怎么调都不准」的真正根因——先看入库产物再调检索。

### 4.3 Embedding 与向量化

**高频问法**
- 「Embedding 模型怎么选的？为什么用 bge-m3？」
- 「1024 维和 3072 维有什么实际差别？维度越高越好吗？」
- 「Embedding 模型能直接算两段话相似度吗？和 Reranker 什么区别？」
- 「换了 Embedding 模型，历史向量怎么办？」
- 「查询和文档用同一个模型编码吗？检索指令（instruction）是什么？」

**答题要点**
- 选型维度：语言（中文优先 bge-m3 / 国产系）、部署形式（本地私有化 vs API）、是否需要稀疏 + 稠密混合（bge-m3 原生支持 dense + sparse + multi-vector）、许可证。MTEB 榜单版本不一，**自己场景必须重测**。
- 维度是效果 / 存储 / 检索速度的三方权衡；部分模型支持 Matryoshka 降维。
- Bi-encoder（Embedding，双塔分开编码，可预索引，快）vs Cross-encoder（Reranker，拼接交互，准但慢 100 倍）——「召回用双塔、精排用交叉」是标准答案。
- 换模型 = 全量重建索引：向量空间不同不能混用；生产做法是双索引灰度切换，评测通过后下线旧索引。
- 检索指令细节：bge-v1.5 系检索任务要求给 query 加指令前缀（「为这个句子生成表示以用于检索相关文章：」），漏加会掉点；**bge-m3 官方明确不需要任何指令前缀**——用哪个模型就核对哪个的模型卡，这类细节答对说明真用过。

### 4.4 向量数据库与检索

**高频问法**
- 「pgvector、Milvus、Qdrant、ES 怎么选？你们为什么用 pgvector？」
- 「HNSW 和 IVFFlat 的原理和取舍？」
- 「向量检索是精确的吗？ANN 的召回率怎么保证？」
- 「百万级向量用什么方案？亿级呢？」
- 「多租户 / 权限过滤怎么做？先过滤还是先检索？」

**答题要点**
- 选型逻辑：数据量十万~百万级、已有 PostgreSQL、想统一持久层和事务 → pgvector（少一个组件，运维成本低，能和业务表 join 做权限过滤）；亿级、高 QPS、需要分布式 → Milvus；轻量云原生 → Qdrant；已有 ES 且要全文 + 向量一体 → ES。**答出「为当前规模选择，不为想象的规模过度设计」是加分**。
- HNSW：多层跳表式图索引，查询 log 级、内存大、建索引慢、支持增量；IVFFlat：聚类分桶，内存小、建快、召回对 nprobe 敏感且数据分布变化后要重建。参数（m / ef_construction / ef_search）影响召回率与延迟。
- 权限过滤必须讲：前置过滤（filter 下推到索引，pgvector 用 WHERE + 部分索引）优于后置过滤（检索完再筛，可能筛空）；多租户按 owner_id 隔离是数据安全红线（IDOR 防护延伸到知识库）。

### 4.5 混合检索与 Rerank

**高频问法**
- 「为什么要 BM25 + 向量混合检索？各自补什么短板？」
- 「RRF 融合的公式和直觉？为什么不用分数加权平均？」
- 「Rerank 解决什么问题？为什么 Cross-Encoder 精度高这么多？」
- 「召回 top 多少送 Rerank？Rerank 后取多少？延迟加了多少能接受吗？」
- 「中文混合检索有什么特殊处理？」

**答题要点**
- 互补性：向量擅长语义近似（「报销」≈「费用申请」），BM25 擅长精确词 / 专有名词 / 型号数字（Embedding 会把它们压糊）；混合召回率通常比单路高 5~15 个百分点。
- RRF：`score = Σ 1/(k + rank_i)`，只用名次不用分数，天然规避两路分数量纲不可比的问题，k（默认 60）调节头部权重；加权平均需要归一化和调参，鲁棒性差。
- Rerank 工程账：召回 top 50~100 → Cross-encoder（如 bge-reranker-v2-m3）精排 → top 3~10 进上下文；延迟加几十到几百毫秒，换精度大幅提升；**Rerank 救不了没召回的文档**——召回率优先，精度靠精排兜底。
- 中文特殊性：BM25 依赖 jieba 等分词；领域术语要词典补强；中文场景 BM25 权重通常要调高。

### 4.6 Query 理解与改写

**高频问法**
- 「用户问题很口语化 / 有错别字 / 指代不清（"那这个呢？"），检索前怎么处理？」
- 「Query 改写有哪些策略？多轮对话里的 Query 怎么补全？」
- 「HyDE 是什么原理？什么时候有用？」
- 「改写用大模型做，延迟和成本怎么办？」

**答题要点**
- 策略工具箱：指代消解（结合对话历史补全）、标准化 / 纠错、同义扩展（多 Query 并行检索取并集）、复杂问题分解（多跳拆子问题）、HyDE（先生成假设答案再拿去检索，弥合问答语义鸿沟）。
- 工程取舍：改写加一次 LLM 调用（延迟 + 成本），用小模型 / 低 max_tokens 做；简单 Query 可用规则旁路跳过改写。
- 意图识别前置：先判断「要不要检索」（闲聊直接答，业务问题走 RAG），避免全量过重管线。

### 4.7 生成阶段与引用溯源

**高频问法**
- 「检索到的内容怎么组织进 Prompt？顺序有讲究吗？」
- 「怎么让答案带引用？引用怎么定位到原文？」
- 「检索结果为空或质量很差时，直接让模型自由发挥吗？」
- 「用户追问时还要重新检索吗？」

**答题要点**
- 上下文组织：带来源标记（[1][2]）注入、关键证据放头尾（抗 Lost in the Middle）、去重合并同文档相邻块。
- 引用溯源：要求模型输出句级 / 段级引用标记，后处理映射回 chunk → 文档 → 页码，前端渲染可点击来源（SSE 里作为独立 reference 事件推送）。
- 空召回兜底：明确告知「知识库中未找到」，可降级通用回答但要打标区分，禁止把无证据回答伪装成检索结果。
- 多轮 RAG：改写补全后每轮重检索，或按意图判断是否复用上轮证据。

### 4.8 RAG 效果排查与优化（必考场景题）

**高频问法**
- 「上线后用户反馈答非所问，你从哪开始查？」（最高频 RAG 场景题）
- 「召回率低怎么优化？给一个排查顺序。」
- 「知识库更新了，旧答案还在返回，可能是哪些原因？」
- 「你做过哪些优化？每一项带来多少提升？」

**标准排查路径（背下来）**
1. **先归因到段**：拿 bad case 看检索日志——正确证据在不在召回候选池里？
2. **不在池里**（召回问题）：查文档是否入库成功 → 解析产物对不对 → 切片是否切碎语义 → Embedding 是否适配领域词 → 元数据过滤是否误杀 → 上 Query 改写 / 混合检索。
3. **在池里但排名靠后**（排序问题）：上 / 调 Rerank → 扩召回窗口 → 调 RRF 权重或候选池大小。
4. **进了上下文答案还错**（生成问题）：查 Prompt 约束、证据在上下文中的位置、模型忠实度（Faithfulness 评测）、考虑换模型或加引用强约束。
5. **固化**：修复后回灌评测集，防止复发。

**知识库更新不生效的排查**：入库任务是否异步完成（事件触发 + 补偿任务）→ 旧 chunk 是否删除（软删 / 版本标记）→ BM25 内存索引是否重建（多 worker 各有缓存，需要版本号广播失效）→ 语义缓存是否还命中旧答案（TTL / 主动失效）。

### 4.9 进阶 RAG 范式（2026 新热点）

**高频问法**
- 「GraphRAG 解决什么问题？代价是什么？」
- 「Agentic RAG 和普通 RAG 的区别？什么场景值得上？」
- 「Self-RAG / CRAG 的思路说说看。」
- 「多模态文档（图表 / 图片）怎么进 RAG？」

**答题要点**
- GraphRAG：LLM 抽取实体关系建图 + 社区摘要，解决**跨文档多实体全局性问题**（「总结所有项目的共同风险」）；代价是入库成本大涨（每 chunk 过 LLM），增量更新难——大多数场景先把朴素 RAG + Rerank 做扎实。
- Agentic RAG：LLM 自主决定检索次数与策略（改写 → 检索 → 反思 → 再检索），解决多跳推理；成本与延迟不可控，需限最大轮数。
- Self-RAG（模型自判何时检索、检索是否相关）、CRAG（对召回质量打分，低分触发纠正 / 联网搜索）——答出「解决召回质量参差」即可。
- 多模态：图表用 VLM 生成描述后入库，或多模态 Embedding 统一空间；面试重点是**成本与收益的判断**而非堆概念。

## 五、Agent、工具调用与 MCP

> 考察定位：2026 年 Agent 已从「加分项」变成「主考区」。面试官验证三件事：你理解 Agent 与 Workflow 的本质区别吗？你的 Agent 失控时怎么办（步数 / 成本 / 死循环）？工具边界与权限治理做了吗？

### 5.1 Agent 基础与 ReAct

**高频问法**
- 「什么是 Agent？和 Chatbot、和写死的工作流有什么本质区别？」
- 「ReAct 的循环是怎么跑的？Thought / Action / Observation 各是什么？」
- 「你们的 Agent 最多跑多少步？为什么要限制？」
- 「ReAct 和 Plan-and-Execute 的取舍？」
- 「什么任务不该用 Agent？」（反向题，高频）

**答题要点**
- 本质区别是**控制流归属**：Workflow 由代码编排（if/else/DAG），路径可预测；Agent 由 LLM 动态决定下一步和工具使用，路径不可预测。因此 Agent 的成本、延迟、调试难度都高一档。
- ReAct 循环：模型输出思考 + 工具调用 → 执行工具 → 观察结果回填 → 再决策，直到产出最终答案；实现上是 LangGraph 的 agent 节点 + ToolNode 条件循环。
- 必答的护栏：**最大步数**（如 10 步）、单任务 Token / 成本预算、超时、重复失败检测（同一工具同参数连续失败即跳出）、终态兜底（超限后用已有信息给最佳回答）。
- 反向题标准答案：**能用 Workflow 就不用 Agent**——步骤可预知、确定性要求高、错误成本高的任务写死流程；只有步数不可预知 + 需要环境交互的任务才值得 Agent（Anthropic《Building Effective Agents》立场）。

### 5.2 工具（Tool）设计与治理

**高频问法**
- 「工具的 Schema 怎么写才能让模型选得对、传参传得对？」
- 「模型幻觉出不存在的参数 / 选错工具怎么办？」
- 「工具调用失败，Agent 层面怎么处理？」
- 「危险操作（发通知、写数据、转账）怎么管控？」
- 「工具返回一大坨 JSON，直接塞回上下文吗？」

**答题要点**
- Schema 即 Prompt：一句话用途 + 每个参数的类型 / 含义 / 示例 + 返回格式 + 何时不该用这个工具；工具粒度小而正交（可组合），命名反映业务语义。
- 防误用：枚举约束参数、Pydantic 校验入参、错误信息写成「模型能自我纠正的提示」（"日期格式应为 YYYY-MM-DD，你传的是…"）而不是裸异常码。
- 失败处理分层：工具内部重试（网络类）→ 把失败作为 Observation 交给模型换路 → 连续失败熔断该工具并告知模型 → 整体降级。
- 危险操作治理：读写分离、写操作显式确认（Human-in-the-Loop）、deny-first 权限白名单、幂等键防重放、审计日志；**未接入的能力不给模型暴露工具**（避免模型假装执行成功）。
- 大结果处理：截断 + 摘要 + 写外部存储留引用，防上下文爆炸。

### 5.3 Function Calling、MCP 与协议生态（A2A / Agent Skills）（2026 必考）

**高频问法**
- 「MCP 是什么？和 Function Calling 是什么关系？谁替代谁？」
- 「MCP 的三种原语（Tools / Resources / Prompts）分别是什么？」
- 「为什么说 MCP 解决了 N×M 问题？」
- 「生产环境接入第三方 MCP Server，你担心什么？怎么治理？」
- 「写过 MCP Server 吗？stdio 和 HTTP（Streamable HTTP）传输怎么选？」
- 「A2A 协议是什么？和 MCP 什么关系？」（2026 新增）
- 「Agent Skills 听说过吗？和 Tool / MCP 的分工是什么？」（2026 新增）

**答题要点**
- 关系不是替代：Function Calling 是**模型厂商 API 的能力**（模型输出调用意图）；MCP 是**应用与工具生态之间的开放协议**（统一发现、调用、鉴权外部工具 / 资源）。MCP 客户端拿到工具清单后，最终仍通过 Function Calling 让模型选择工具。
- N×M → N+M：以前每个应用为每个工具写适配器；MCP 统一协议后工具实现一次、处处可用（Host–Client–Server 三层）。
- 生产治理（区分度所在）：工具描述可信吗（恶意 Server 可做提示注入）→ 权限最小化（只挂需要的工具、读写分级）→ 输入输出审计 → 敏感数据不出域 → 版本锁定与供应链审查 → 破坏性工具强制确认。
- 开发细节：stdio 适合本地进程（注意**禁止向 stdout 打日志**，会破坏协议帧）；远程服务用 Streamable HTTP + OAuth；工具要有清晰错误分层与分页设计。
- A2A（Agent2Agent）：**MCP 管「Agent→工具」，A2A 管「Agent→Agent」**，互补分层不竞争。Google 2025-04 推出、已捐给 Linux Foundation 治理、2026 年发布 v1.0（生产可用）；核心机制是 Agent Card（`/.well-known/` 下的能力名片，供发现与鉴权）+ 标准化任务委托生命周期。答题口径：单 Agent 项目 MCP 够用，跨团队 / 跨厂商 / 跨组织的多 Agent 协作才引入 A2A。
- Agent Skills：把领域知识与操作手册打包成可复用模块（SKILL.md 开放格式），核心是**渐进式披露**三级加载——元数据（name + description）常驻上下文、正文触发时载入、脚本与参考资料按需读取，几十个技能常驻也只占极少 Token。分工一句话：**Tool / MCP 给能力（手），Skill 给方法与知识（脑）**。

### 5.4 Agent 失控与稳定性（高区分度）

**高频问法**
- 「Agent 死循环烧 Token 的事故听过 / 遇到过吗？怎么防？」
- 「Agent 跑到一半上下文爆了怎么办？」
- 「怎么让 Agent 不再犯同样的错误？」（Harness 思维题）
- 「Agent 的执行过程怎么让用户看到？怎么让开发者能调试？」

**答题要点**
- 死循环模式：失败重试同一调用、目标漂移原地打转、格式幻觉死磕、多 Agent 互相否定。防御纵深：硬性步数上限 + 超时 + Token 预算 + 进度检测（连续 N 步无状态变化强停）+ 重复失败检测 + 成本告警。
- 上下文管理：工具大输出折叠、阶段性压缩摘要、长任务把计划与进度写外部状态（文件 / Redis），跨会话可续跑。
- Harness 思维（2026 加分）：Agent 犯错后不是「希望它下次好点」，而是**工程化封堵**——能用确定性约束（校验、lint、权限、CI 硬失败）就不用提示词；错误信息本身设计成可自愈的 Prompt。
- 可观测：每步作为 SSE 事件流式展示（step / tool_call / tool_result / answer）；后端全链路 trace（每次 LLM 调用的 prompt / 响应 / token / 耗时 / 成本可回放）。

## 六、多 Agent 与工作流编排（LangGraph）

> 考察定位：考你「什么时候拆多 Agent、怎么编排、状态怎么管」。用 LangGraph 的团队会深挖 State / Checkpoint / 中断恢复；没用过框架的会被问手写思路。

### 6.1 LangGraph 核心机制

**高频问法**
- 「为什么用 LangGraph 而不是裸写循环 / 用 LangChain AgentExecutor？」
- 「StateGraph 的 State 怎么设计？add_messages 这种 reducer 是干嘛的？」
- 「条件边（conditional edges）怎么用？你们的图长什么样？」
- 「Checkpoint / thread_id 是什么机制？怎么实现『断点续跑』和『人工审批后继续』？」
- 「LangGraph 的 interrupt（人机协同）怎么做的？」

**答题要点**
- 选型理由：显式图结构（可视化、可测试）、状态机语义（State + Reducer 合并更新）、原生支持循环 / 分支 / 并行、Checkpoint 持久化（中断恢复、时间旅行）、SSE 流式事件（astream_events）。
- State 设计：TypedDict + Annotated reducer（如 add_messages 追加消息而非覆盖）；只放必要状态，大对象放外部存储留引用。
- 中断恢复标准答案：编译时挂 checkpointer（生产用 PostgreSQL / Redis 持久化，不用内存版）→ 以 thread_id 标识一次运行 → interrupt 节点暂停等待人工输入 → 恢复时同 thread_id + resume 值继续；并发恢复要加分布式锁仲裁（同一 thread 同时只能有一个执行者）。
- 停用配置的语义边界（真实业务细节，强加分）：停用工作流阻止**新任务启动**，不追溯取消已受理 / 暂停的任务——运行中任务按快照继续或由用户显式取消。

### 6.2 多 Agent 协作模式

**高频问法**
- 「什么时候值得拆多 Agent？拆错了有什么代价？」
- 「Supervisor / 层级式 / Swarm 三种拓扑怎么选？」
- 「Agent 之间怎么通信？共享什么、隔离什么？」
- 「多 Agent 吵起来 / 互相推翻怎么办？」
- 「你们的多级审批模拟是怎么编排的？」

**答题要点**
- 拆分判断：任务有清晰角色分工（如审批链的不同审批人视角）、子任务上下文差异大（避免单上下文互相污染）、需要并行才拆；**多 Agent 不是银弹**——成本、延迟、调试复杂度全部上升，简单任务拆了是负收益。
- 拓扑选择：Supervisor（主控拆分派发合并，决策集中易调试，生产首选）；层级式适合长程复杂任务；Swarm（去中心化）适合探索性任务但难收敛。避免「民主投票群聊」反模式。
- 通信与状态：结构化消息（角色 / 输入 / 输出契约）、共享状态单写多读、每个 Agent 独立上下文窗口，主 Agent 只看摘要。
- 终止与仲裁：最大轮数、仲裁者角色、明确验收标准（谁说了算），防止互相否定死循环。

### 6.3 SSE 流式 + 长任务工程化

**高频问法**
- 「工作流跑几分钟，HTTP 请求早超时了，怎么设计交互？」
- 「SSE 推到一半服务重启 / 用户刷新页面，任务状态怎么办？」
- 「多个 worker 实例下，恢复请求路由到另一台机器还能续上吗？」

**答题要点**
- 模式：受理即返回 run_id（任务受理与执行解耦）→ SSE 订阅进度事件 → 断线按 run_id 重连拉状态快照续流。
- 状态外置：运行快照写 Redis（带 TTL），任何 worker 都能按 thread_id 恢复；恢复加分布式锁防并发双跑；LangGraph checkpointer 保证图内状态一致。
- 终态语义：完成 / 失败 / 取消都要落终态事件与持久化记录，禁止「流断了用户猜结果」。

## 七、记忆系统（Memory）

> 考察定位：从「多轮对话怎么带历史」到「长期个性化记忆怎么设计」，逐层加深。

**高频问法**
- 「多轮对话的上下文怎么管理？全部塞进去吗？」
- 「短期记忆和长期记忆怎么划分？各自存哪？」
- 「滚动摘要怎么做？摘要丢了关键信息怎么办？」
- 「用户画像 / 偏好这类跨会话记忆怎么抽取、怎么更新、怎么防止污染？」
- 「Agent 长任务的『工作记忆』怎么设计？」
- 「了解 mem0 / Letta（MemGPT）/ Zep 这类记忆组件吗？」

**答题要点**
- 短期记忆：最近 N 轮原文 + 更早轮次滚动摘要（summary buffer）；按 Token 预算裁剪而非按条数；持久化到 PostgreSQL（会话历史是业务数据，不是缓存）。
- 长期记忆：结构化画像（姓名 / 偏好 / 业务属性，LLM 结构化抽取 + 字段级合并更新）与向量化情景记忆（历史对话切片入向量库按需检索）两条路线；写入要有置信度门槛与来源标记，防止一句玩笑话污染画像（记忆污染）。
- 分类速答：工作记忆（当前上下文窗口）/ 情景记忆（历史事件，向量检索）/ 语义记忆（事实与画像，结构化存储）/ 程序记忆（技能与规则，Prompt / 代码）。
- Agent 长任务：计划与进度写外部文件 / 状态存储（progress 文件思路），新会话开机先读进度——上下文会丢，外部状态不丢。
- 遗忘与更新：TTL、重要性打分淘汰、同键新值覆盖 + 冲突时间戳仲裁。
- 开源记忆组件速答：MemGPT / Letta 是 OS 式分层记忆（主上下文 + 外部存储换页）、mem0 是「抽取 → 冲突消解 → 存储 → 检索」流水线、Zep 偏时序知识图谱——答出共同机制（写入抽取、检索注入、更新仲裁、遗忘淘汰）比背产品名重要。

## 八、模型层与推理部署（进阶加分）

> 考察定位：应用岗不强制手推 Transformer，但 3 年以上或大厂岗会抽查「你知道你调用的东西是怎么跑起来的吗」。答出工程含义即可，不需要论文级推导。以下按「应用岗必答 / 进阶加分」分级。

### 8.1 架构与推理原理（应用岗必答部分）

**高频问法**
- 「Transformer 为什么能并行训练？和 RNN 的本质区别？」
- 「KV Cache 是什么？为什么长对话越聊越慢、显存越占越多？」
- 「LLM 推理为什么分 Prefill 和 Decode 两阶段？瓶颈有什么不同？」
- 「首 Token 延迟（TTFT）和 Token 间延迟（TBT）分别由什么决定？」

**答题要点**
- Self-Attention 全序列并行计算 vs RNN 顺序依赖；位置信息靠位置编码（RoPE 主流，具备一定外推性）。
- KV Cache：缓存历史 Token 的 K/V 避免重复计算，显存占用随序列长度线性涨——这是长上下文贵与慢的底层原因，也是 Prompt 前缀缓存（同前缀复用 KV）的原理。
- Prefill 吃算力（并行处理整段输入，决定 TTFT），Decode 吃显存带宽（逐 Token 生成，决定 TBT）——所以「输入长影响首字延迟，输出长影响总时长」，PD 分离部署由此而来。

### 8.2 训练与微调（考「什么时候需要微调」的判断）

**高频问法**
- 「预训练 / SFT / RLHF（对齐）各阶段做什么？」
- 「LoRA 的原理？为什么能省这么多显存？」
- 「什么情况该微调而不是 RAG / Prompt？你会怎么准备数据？」
- 「DPO 和 PPO 的区别？听说过 GRPO 吗？」（大厂进阶）

**答题要点**
- 判断题标准答案：先 Prompt → 再 RAG → 最后才微调（改风格 / 领域语言 / 稳定格式，且有几千条以上高质量数据时）；微调解决「行为」，RAG 解决「知识」。
- LoRA：冻结原权重，训练低秩增量矩阵（W + BA），可插拔、多任务共存；QLoRA 再加 4bit 量化基座。
- 对齐简版：SFT 教格式与指令跟随；RLHF/DPO 教偏好；GRPO（DeepSeek R1 使用）用组内相对奖励省掉 Critic 模型——答出「R1 用 GRPO + 可验证奖励涌现推理能力」即是加分上限。

### 8.3 推理加速与私有化部署（2026 热度上升）

**高频问法**
- 「vLLM 为什么快？PagedAttention 解决什么问题？」
- 「Continuous Batching 和静态 Batching 的区别？」
- 「量化（INT8 / INT4 / FP8）对效果和吞吐的影响？怎么选？」
- 「私有化部署一个 32B 模型要多少显存？怎么估算？」
- 「MoE 是什么？为什么 DeepSeek 又大又便宜？」
- 「PD 分离（Prefill / Decode 分离）是什么？为什么大规模推理服务都在做？」

**答题要点**
- PagedAttention 把 KV Cache 分页管理（类操作系统虚拟内存），消灭显存碎片，配合 Continuous Batching（请求随到随进随出，不等整批）大幅提升吞吐。
- 显存估算：权重（参数量 × 精度字节，如 32B FP16 ≈ 64GB）+ KV Cache（随并发与序列长度）+ 激活开销；INT4 量化权重减 4 倍，效果小损失换部署可行性。
- MoE：每 Token 只激活部分专家（DeepSeek-V3 671B 总参 / ~37B 激活），推理成本按激活算——「总参数大 ≠ 推理贵」；MLA（多头潜在注意力）压缩 KV Cache 是 DeepSeek 便宜的另一半原因（KV 缓存小到可以放硬盘，这也是其上下文硬盘缓存便宜的底层支撑）。
- PD 分离（2026 新热点）：Prefill 吃算力、Decode 吃显存带宽，混部时长 Prompt 的 Prefill 会挤占同卡请求的 Decode（TBT 抖动、SLA 互相拖累）；分离部署把两阶段放进独立实例池（可按各自瓶颈配不同硬件），KV Cache 经高速互联传给 Decode 池，TTFT 与 TBT 得以独立扩缩与优化——大规模推理服务的标配。代价是 KV 传输开销与调度复杂度，小规模自部署单机 vLLM 足矣——答出「什么规模才值得」即体现判断力。
- 进阶追问再深一层：FlashAttention（IO 感知的精确注意力，减少 HBM 读写而非近似计算）、投机解码（小模型出草稿 + 大模型并行验证，降延迟不降质量，适合输出长 / 流量大的场景）——答出「优化的是访存而不是算法复杂度」即可。

## 九、AI 系统设计（社招必考）

> 考察定位：开放题，考架构分层与治理闭环。答题套路：**先问清需求边界（规模 / 延迟 / 合规）→ 画分层架构 → 逐层讲关键决策 → 主动讲质量闭环（评测 / 观测 / 灰度）→ 留出演进路线**。

### 9.1 设计一个企业级知识库问答系统（出现率最高）

**面试官期待的模块清单**
1. **接入层**：鉴权（JWT）、限流、SSE 网关。
2. **入库管线**：上传 → 解析 → 切片 → 向量化 → 索引，事件驱动异步化 + 失败补偿任务 + 分布式锁防并发重复处理；文档状态机（处理中 / 就绪 / 失败）。
3. **检索层**：意图识别 → Query 改写 → 混合检索（BM25 + 向量 RRF）→ Rerank → 权限过滤（owner / 部门隔离，**先过滤后检索**）。
4. **生成层**：Prompt 模板管理、引用溯源、流式输出、兜底话术。
5. **治理层**：评测集 + CI 门禁、全链路 Trace、Token 成本归因、预算护栏、缓存（精确 + 语义）。

**高频追问**
- 「文档更新后多久生效？怎么保证旧内容不再被检索到？」→ 版本化 chunk + 原子切换 + 缓存失效 + BM25 索引重建广播。
- 「不同部门的人不能搜到彼此的文档怎么做？」→ 元数据权限过滤下推到检索层，接口层二次校验，测试覆盖越权用例。
- 「百万文档、千 QPS 怎么扩展？」→ 检索层无状态水平扩、向量库换分布式 / 分片、Embedding 服务独立部署批量化、热点语义缓存。

### 9.2 设计一个生产级 Agent 平台

**要点**：任务受理与执行解耦（run_id + 状态机：受理 / 运行 / 暂停 / 终态）、工具注册与权限矩阵（deny-first）、步数与预算护栏、Checkpoint 持久化与断点恢复、人机协同（审批中断）、执行轨迹可回放、多租户隔离与配额。

**高频追问**：「Agent 执行到一半服务发版怎么办」（快照 + 恢复锁）；「怎么防止一个用户把预算烧光」（租户级预算原子预留 + 结算，Redis Lua）；「工具生态怎么开放给其他团队」（MCP 标准化 + 审核上架流程）。

### 9.3 设计模型网关（LLM Gateway）

**要点**：统一接入多供应商（协议适配）→ 路由（按任务难度 / 租户 / 区域，简单任务走小模型）→ Fallback 链（5xx / 429 / 超时切备用，跨厂商兜底）→ 双层缓存（精确哈希 + 语义相似，语义阈值保守防误命中）→ 多维限流（租户 × 模型，RPM + TPM 双桶）→ 计量计费（逐请求记录 token 与成本，按业务归因）→ SSE 透传不缓冲 → 全链路 Trace。

**高频追问**：「语义缓存误命中造成答非所问怎么办」（高阈值 + 离线验证 + 可关闭）；「供应商悄悄换模型版本导致效果漂移怎么发现」（锁 model 版本字段 + 每日黄金集回归）。

### 9.4 设计评测体系 / 内容安全网关 / Text2SQL / 实时语音 Agent（次高频）

- **评测体系**：见第十章，答「三层组合 + CI 门禁 + bad case 回灌」。
- **内容安全**：输入输出双向过滤（规则 + 小模型分类器 + 敏感词），流式输出的增量审核（按句缓冲），审计留痕，误杀与漏放的阈值权衡。
- **Text2SQL / 结构化数据问答**（企业数据分析场景高频）：链路 = Schema 裁剪注入（只给相关表 + 字段注释 + 少量示例行）→ LLM 生成 SQL → **执行前校验**（语法解析、只允许 SELECT、表白名单、LIMIT 强制、EXPLAIN 预估成本）→ 沙箱只读账号执行 + 超时 → 错误回填重试自愈 → 结果摘要生成。安全红线：连接账号只读最小权限、行级权限过滤不能交给 LLM 生成的 WHERE 来保证；效果关键：表 / 字段的业务语义注释质量 > Prompt 技巧。
- **实时语音 Agent**：ASR → LLM → TTS 全链流式化、双工打断（barge-in：VAD 检测到用户说话立即停 TTS 并取消生成）、端到端延迟预算分解（目标 < 1s）、对话状态机；答出「延迟是这个场景的第一约束」即抓住重点。

## 十、评测与质量保障

> 考察定位：2026 年区分「玩具项目」和「生产项目」的最强信号。没有评测集，你的一切优化都是「我感觉变好了」。

### 10.1 离线评测与 RAGAS

**高频问法**
- 「你们怎么评测 RAG 效果的？指标有哪些？」
- 「RAGAS 的四个核心指标分别度量什么？怎么算的？」
- 「评测集（Golden Set）怎么构建？多少条够用？覆盖哪些类型？」
- 「检索和生成怎么分开评？」

**答题要点**
- RAGAS 经典四指标：**Faithfulness**（答案的每个论断是否被检索上下文支撑 = 可验证论断 / 全部论断，度量编造）；**Answer Relevancy**（答案是否切题：用答案反推出若干问题、与原问题算 Embedding 相似度，**不需要标准答案**）；**Context Precision**（top-k 里相关块占比与排位，度量检索噪声）；**Context Recall**（标准答案所需信息被检索覆盖的比例，度量漏召回）。注意 **Factual Correctness**（答案与标准答案做论断级比对算 F1）是**另一个需要 ground truth 的独立指标**，别与 Answer Relevancy 混为一谈——「怎么算的」这层追问验的就是这里。常用阈值 ≥ 0.7 起步，Faithfulness 优先修。
- Golden Set 构建：50~1000 条，来源 = 真实用户问题采样 + 领域专家出题 + 对抗样本；覆盖四类查询（事实型 / 多跳型 / 汇总型 / 无答案型——「无答案型」考拒答能力，最容易被漏）。人工标注标准答案与相关文档，**版本化管理，私有不外泄**。
- 分段评测：检索层用 Recall@K / MRR / NDCG（不依赖 LLM，快且稳定）；生成层用 RAGAS / LLM-as-Judge；先分段归因再看端到端。

### 10.2 LLM-as-Judge

**高频问法**
- 「用 LLM 评 LLM 靠谱吗？有哪些偏差？怎么缓解？」
- 「Judge 模型怎么选？能用被评模型自己当裁判吗？」
- 「Pointwise 打分和 Pairwise 对比怎么选？」
- 「怎么校准 Judge 和人工判断的一致性？」

**答题要点**
- 三大偏差与缓解：位置偏差（交换 A/B 两次评，不一致记平）、冗长偏差（rubric 拆维度打分、明示不受长度影响）、自偏好（换家族模型当 Judge、多 Judge 投票）。
- 判官原则：Judge 能力应 ≥ 被评模型；确定性检查（格式 / 关键词）用规则不用 LLM。
- 选型：模型选型用 Pairwise（一致性高），生产监控用 Pointwise + rubric（可上看板）。
- 人工校准：定期抽样人工复核，算一致率（目标 > 0.7~0.8），低了就修 Judge Prompt——**Judge Prompt 本身也要版本化和评测**。

### 10.3 CI 门禁与线上质量闭环

**高频问法**
- 「Prompt / 检索参数改了，怎么保证不劣化线上效果？」
- 「评测怎么集成进 CI？阈值怎么定？评测太慢太贵怎么办？」
- 「上线后怎么持续发现 bad case？发现后的闭环是什么？」
- 「LLM 调用不稳定，评测测试老是波动怎么办？」

**答题要点**
- CI 门禁：核心评测集跑分低于阈值阻断合并；分层执行——单测 mock LLM（禁真实调用，mock 国产模型返回值）、评测集用小样本冒烟 + 全量夜跑；用 pytest marker 分离（unit / evaluation / live）。
- 波动治理：固定采样参数与模型版本、多次运行取均值、设置显著性带宽而非单点阈值。
- 线上闭环：全链路 Trace（Langfuse 等）+ 用户反馈（点赞点踩）+ 自动异常指标（拒答率 / 长度突变 / 工具失败率）→ bad case 自动入库 → 每周归因 → 修复 → **回灌评测集**（Harness 闭环：同类错误永不复发）。
- A/B 与漂移：按 user hash 稳定分桶；输入分布 / 输出分布 / 每日黄金集跑分三路漂移监控。

### 10.4 Agent 评测（2026 新热点）

**高频问法**
- 「RAG 有 RAGAS，那 Agent 怎么评？和评一次问答的本质区别是什么？」
- 「只看任务最后成没成功够吗？中间过程要不要评？怎么评？」
- 「pass@k 和 pass^k 的区别？为什么生产 Agent 更该看 pass^k？」
- 「τ-bench / SWE-bench / GAIA 这些基准了解吗？对你自己的业务 Agent 有什么参考价值？」
- 「Agent 评测环境怎么搭？工具和外部系统怎么处理？」

**答题要点**
- 本质区别：问答评「一个输出」，Agent 评「一条轨迹」——多步、有状态、正确路径不唯一、失败可能发生在任意一步，所以必须**结果 + 过程**两层评，缺一不可。
- 结果层：任务成功率用**可验证的终态断言**判定（数据库状态、文件产物、环境检查），不能信模型自述「已完成」；pass@k（k 次至少成 1 次，测能力上限）vs pass^k（k 次全部成功，测稳定性）——用户要的是「每次都对」，生产 Agent 盯 pass^k，它随 k 增大的衰减暴露的正是不稳定性。
- 过程层（轨迹评测）：工具选择正确率、参数正确率、步数效率（冗余步 / 原地打转）、违规动作检测（越权调用、跳过确认）；轨迹与参考轨迹做匹配（精确 / 顺序 / 无序分级，容忍等价路径），或对每步用 LLM-as-Judge 按 rubric 打分（这一步是否朝目标推进），Judge 结论定期人工校准（同 10.2）。
- 评测环境工程：环境可重置、工具走 mock / 沙箱（固定返回、禁真实副作用）、主动注入失败场景（工具报错 / 超时 / 空结果）考察恢复能力；对话型任务用 LLM 模拟用户多轮交互（τ-bench 思路）。
- 公开基准速答：τ-bench（模拟用户 + 业务策略遵守，客服域，pass^k 出处）、SWE-bench（真实 GitHub issue 修复）、GAIA（通用助手多步任务）——答题口径：「公开基准看模型选型趋势，业务 Agent 必须自建任务级评测集」，把线上 bad case 转成回归用例。
- 闭环同 10.3：小样本冒烟进 CI、全量夜跑；轨迹全量落 Trace 可回放，bad case 回灌评测集。

**扣分点**：只答「看任务成功率」；不知道轨迹 / 过程评测；把 pass@k 当稳定性指标；评测环境直连真实系统产生副作用。

## 十一、安全、合规与护栏

> 考察定位：企业级场景的一票否决区。国内面试还会叠加合规问法（生成式 AI 备案、数据出境）。

### 11.1 OWASP LLM Top 10 视角

**高频问法**：「LLM 应用特有的安全风险有哪些？和传统 Web 安全的区别？」

**答题要点**（2025 版十项）：Prompt 注入（LLM01）、敏感信息泄露（LLM02，从旧版第 6 升至第 2）、供应链（LLM03，第三方模型 / MCP Server / 依赖）、数据与模型投毒（LLM04）、不安全输出处理（LLM05，LLM 输出直接进 SQL / HTML / shell = 新型注入源）、过度代理（LLM06，Agent 权限过大）、**系统提示泄露（LLM07，2025 新增）**、**向量与嵌入弱点（LLM08，2025 新增，RAG 专属）**、错误信息（LLM09）、无界消耗（LLM10，Token 轰炸 / 死循环烧钱）。

两个 2025 新增项最值得展开（都指向 RAG / Agent 生产化教训）：
- LLM07 系统提示泄露：假设 System Prompt 一定会被套出来——**密钥、权限逻辑、业务规则绝不放 Prompt**，安全控制放模型外的确定性系统（鉴权中间件、权限层），Prompt 不是安全边界。
- LLM08 向量与嵌入弱点：向量库越权访问、多租户共享嵌入空间导致数据串租、投毒文档在检索时触发间接注入、嵌入反演还原原文——对策是向量存储按租户严格分区、入库源校验、检索结果消毒后再入上下文。

### 11.2 Prompt 注入与越狱

**高频问法**
- 「直接注入和间接注入的区别？各举个例子。」
- 「RAG 文档、网页、工具返回值里藏指令怎么防？」
- 「防注入的纵深防御怎么做？只靠 Prompt 里写『不要听用户的』行吗？」

**答题要点**
- 直接注入（用户输入覆盖指令）vs 间接注入（外部内容——检索文档 / 网页 / 邮件——携带指令，Agent 时代更危险，因为会串到工具调用）。
- 纵深防御：输入过滤与消毒 → 指令 / 数据结构性隔离（分隔符、消息角色）→ 输出过滤（敏感词 / 格式校验）→ **权限层兜底**（即使被注入，工具也只有最小权限、写操作要确认）→ 检测与审计（注入特征识别、告警）。核心认知：**Prompt 层防御必然可绕过，最后一道防线永远是权限而非措辞**。
- 越狱（角色扮演 / DAN / 多轮诱导 / many-shot）：安全系统提示 + 输出侧分类器 + 会话级风控。
- 护栏工程化组件可以点名：NeMo Guardrails（输入 / 输出 / 对话 / 检索 / 执行五类 rails）、Guardrails AI（结构与内容校验）、Llama Guard 类安全分类器——但要点是分层防御思想，工具只是实现。

### 11.3 数据安全与合规（国内特色问法）

**高频问法**
- 「用户数据能直接发给模型 API 吗？脱敏怎么做？」
- 「多租户数据隔离在 RAG 里怎么落实？」
- 「生成式 AI 上线要做什么合规动作？」（大厂 / 国企常问）

**答题要点**：PII 检测与脱敏（正则 + NER，占位符回填）；密钥走环境变量 / Secret 管理，日志与 Prompt 不落敏感数据；租户隔离贯穿存储（owner_id）/ 检索（过滤下推）/ 接口（鉴权校验资源归属，防 IDOR）；国内合规关键词：生成式 AI 服务备案 / 深度合成标识（AI 生成内容标注）、个保法（最小必要、目的限定）、数据出境评估（这也是「必须用国产模型」的常见企业背景）。

## 十二、Python 后端工程

> 考察定位：AI 应用岗的地基。异步生态（asyncio + FastAPI + SQLAlchemy async）是 2026 年 Python 后端面试的绝对主线，因为 LLM 应用天然 IO 密集 + 流式。

### 12.1 asyncio 与并发模型

**高频问法**
- 「GIL 是什么？为什么 IO 密集用多线程 / 协程，CPU 密集用多进程？」
- 「事件循环怎么工作的？协程什么时候切换？」
- 「async 函数里调了同步阻塞库会发生什么？怎么发现、怎么修？」
- 「create_task 之后不 await 有什么坑？」
- 「asyncio.gather 一个任务挂了，其他任务呢？」
- 「怎么限制并发数？怎么做超时取消？」

**答题要点**
- GIL 限制同一时刻单线程执行字节码：CPU 密集多线程无收益（用多进程 / C 扩展），IO 密集时 GIL 在等待时释放所以线程 / 协程都有效；协程开销远小于线程，单线程可撑上万并发连接。
- 协程只在 `await` 处让出控制权；`await` 之间的代码原子执行——在 async 函数里跑同步重活（requests、大计算、同步 DB 驱动）会**卡住整个事件循环**，所有请求一起变慢。修法：换异步库（httpx / asyncpg）、`asyncio.to_thread`（IO / 小 CPU）、进程池（重 CPU）。排查：p99 集体飙升 + loop 监控（debug 模式慢回调告警）。
- create_task 不持引用可能被 GC、异常被吞：用集合持引用 + done_callback 记日志。gather 默认一个异常会向上抛（其余任务继续跑但结果丢失语义混乱），批量容错用 `return_exceptions=True`；结构化并发用 TaskGroup（3.11+）。
- 并发控制：Semaphore 限流、`asyncio.timeout` 超时、CancelledError 必须重抛否则取消失效。

### 12.2 FastAPI 核心机制

**高频问法**
- 「FastAPI 为什么快？和 Flask / Django 的本质区别？」
- 「`def` 和 `async def` 路由有什么区别？FastAPI 怎么处理同步路由的？」
- 「Depends 依赖注入的原理和好处？yield 依赖是干嘛的？」
- 「Pydantic v2 为什么快？v1 到 v2 有哪些破坏性变化？」
- 「中间件、BackgroundTasks、lifespan 分别什么场景用？」
- 「SSE 接口在 FastAPI 里怎么写？StreamingResponse 有什么坑？」

**答题要点**
- ASGI 异步协议 + Starlette（Web 层）+ Pydantic（校验与文档）；类型注解即契约，自动生成 OpenAPI。
- 同步 `def` 路由会被丢进线程池执行（不卡 loop 但线程数有限），`async def` 里必须全程 await 兼容库，否则「伪异步」卡死全场——这是最爱考的坑。
- Depends：声明式依赖图，天然可测试（override）、可复用（鉴权 / DB 会话 / 分页参数）；`yield` 依赖在响应后执行清理（session 关闭 / 事务提交回滚）。
- Pydantic v2：核心 pydantic-core 用 Rust 重写，校验快数倍到数十倍；v1→v2 破坏性变化要能点名（`validator`→`field_validator`、`class Config`→`model_config`、`.dict()`→`model_dump()`、更严格的类型强制转换策略）。
- lifespan 管理全局资源（连接池、模型加载、Redis 客户端），替代 on_event；后台任务轻量用 BackgroundTasks，可靠异步用任务队列。
- SSE：`EventSourceResponse`（sse-starlette）/ StreamingResponse + 生成器；注意在生成器里捕获异常并发终态事件、感知客户端断连（`request.is_disconnected`）、代理层关缓冲。

### 12.3 SQLAlchemy 2.0 async 与工程分层

**高频问法**
- 「SQLAlchemy 2.0 async 的 Session 生命周期怎么管理？」
- 「MissingGreenlet / 'Session 已关闭' 报错遇到过吗？根因是什么？」
- 「N+1 查询怎么发现、怎么解决？selectinload 和 joinedload 区别？」
- 「事务边界放在哪层？Service 里两个 Repository 操作怎么保证原子性？」
- 「连接池参数（pool_size / max_overflow / pool_recycle）怎么配？」
- 「为什么要 Controller → Service → Repository 分层？」

**答题要点**
- Session 通过 `Depends(yield)` 注入按请求管理，`expire_on_commit=False` 防止 commit 后属性访问触发同步 IO（MissingGreenlet 根因：异步上下文外触发懒加载）。
- N+1：循环里访问关系属性触发逐条查询；解法 selectinload（二次 IN 查询，适合一对多集合）/ joinedload（单条 JOIN，适合多对一）；异步下禁止隐式懒加载，显式声明加载策略。
- 事务：Service 层定义业务原子边界（一个用例一个事务），Repository 内只 flush 不 commit；跨服务一致性用 outbox / 补偿，不开分布式事务。
- 分层价值：路由只做协议与鉴权、Service 承载业务规则可单测、Repository 隔离 SQL 便于换存储与 mock；ORM 实体字段带注释 / comment 是团队协作规范。

### 12.4 认证与鉴权（JWT / 密码存储）

**高频问法**
- 「JWT 和 Session 的取舍？JWT 三段结构分别是什么？」
- 「为什么要 access + refresh 双 token？过期时间怎么定？」
- 「JWT 不是无状态吗，登出和吊销怎么实现？」
- 「refresh token 泄露了怎么办？轮换（rotation）怎么设计？」
- 「密码怎么存？bcrypt 在 FastAPI 异步路由里直接调有什么问题？」

**答题要点**
- JWT = header.payload.signature：签名防**篡改**不防**偷看**（payload 只是 Base64，敏感信息禁止进 payload）；HS256 共享密钥（单体够用）/ RS256 私钥签公钥验（多服务 / 第三方验签）。与 Session 的核心取舍：无状态省查库、易水平扩，但天然难吊销。
- 双 token 分工：access 短命（分钟~小时级），每个请求纯无状态验签；refresh 长命（天级）只用于换新 access，**必须服务端有状态**——签发时带 jti（唯一 ID），用 Redis 记录其状态。
- 轮换与吊销（区分度所在）：refresh 一次一换、旧 jti 用后即废；再次收到已用 jti = 重放信号，吊销该用户整个令牌族并强制重新登录；登出 = 吊销当前 refresh 的 jti（access 可进短黑名单等自然过期）。能量化风险更好：无轮换时 refresh 泄露后整个有效期（如 7 天）内可无限重放。
- 密码存储：bcrypt / argon2 慢哈希 + 盐（禁明文 / MD5 / SHA1）；慢是抗暴破的特性（cost≈12 单次 100~300ms），但在 async 路径**同步调用会卡住整个事件循环**——并发登录时全部请求与 SSE 一起冻结，必须 `await asyncio.to_thread(bcrypt.checkpw, ...)`。这是 12.1「async 里跑同步重活」的最典型实例。
- 工程细节：401（未认证）与 403（已认证无权限）语义分开；开发用「跳过认证」开关必须有生产护栏（`env=production` 且未启用认证 → 启动即失败 fail-fast，防止配置误用全站裸奔）；前端 401 刷新的并发单飞见 14.3。

**扣分点**：说「JWT 加密了所以安全」；答不出任何吊销 / 登出方案；不知道 bcrypt 同步阻塞事件循环的问题。

## 十三、数据库与中间件

> 考察定位：AI 应用把 PostgreSQL（业务 + 向量）和 Redis（缓存 + 运行态）用到什么深度，是后端功底的直接体现。

### 13.1 PostgreSQL（含 pgvector）

**高频问法**
- 「索引什么时候失效？组合索引最左前缀怎么理解？」
- 「EXPLAIN 怎么看？Seq Scan 一定是坏事吗？」
- 「MVCC 是什么？为什么 PostgreSQL 需要 VACUUM？」
- 「事务隔离级别和各自解决的问题？PostgreSQL 默认是哪个？」
- 「pgvector 的 HNSW 索引参数怎么调？向量列和业务列的联合过滤怎么优化？」
- 「Alembic 迁移怎么保证可回滚？生产迁移的流程是什么？」
- 「JSONB 什么时候用？GIN 索引是什么？」

**答题要点**
- 慢查询排查流程：慢日志定位 → EXPLAIN (ANALYZE, BUFFERS) 看真实行数与扫描方式 → 加 / 改索引或改写查询 → 用真实数据量验证。
- MVCC 多版本快照读不阻塞写；死元组靠 VACUUM 回收，长事务阻碍回收导致膨胀——答出「长事务危害」是加分。
- 隔离级别速答：PostgreSQL 默认 **Read Committed**（且任何级别都无脏读）；Repeatable Read 基于快照，连幻读也基本防住（与 MySQL 语义不同，这是常见对比追问）；Serializable 用 SSI 实现，可能抛串行化失败需要应用层重试。
- pgvector：HNSW（m / ef_construction 建索引，ef_search 查询期调召回）；过滤 + 向量排序的组合场景用部分索引 / 先过滤策略；向量列单独表 + 元数据字段冗余是常见设计。
- 迁移纪律：自动生成必须人工审查、可回滚（downgrade 实现并演练）、大表加列避免长锁（分步：加可空列 → 回填 → 加约束）、上线前备份。

### 13.2 Redis

**高频问法**
- 「你们 Redis 都存了什么？为什么这些数据适合放 Redis？」
- 「缓存穿透 / 击穿 / 雪崩的区别和对策？」
- 「分布式锁怎么实现？SETNX 之后进程挂了怎么办？锁误删怎么防？」
- 「Lua 脚本什么时候用？给个你项目里的实际例子。」
- 「限流算法有哪些？滑动窗口怎么用 Redis 实现？」
- 「Redis 挂了你的服务还能用吗？降级方案是什么？」

**答题要点**
- 典型用途分类答：缓存（会话 / 语义缓存，带 TTL）、运行态（工作流快照、任务状态）、原子账本（预算预留与结算）、协调原语（分布式锁、限流桶）。
- 三兄弟：穿透（查不存在的 key → 空值缓存 / 布隆过滤器）、击穿（热点 key 过期瞬间 → 互斥重建 / 逻辑过期）、雪崩（大批同时过期 → TTL 加随机抖动 / 多级缓存）。
- 分布式锁标准答案：`SET key token NX EX ttl`（原子加锁 + 超时兜底）→ 释放用 Lua 校验 token 再删（防误删他人锁）→ 谈到锁续期（watchdog）与 Redlock 争议是进阶。
- Lua 原子性实战例（预算守卫）：读当前累计 → 判断是否超预算 → 预留额度写入，三步必须原子，否则并发下超卖——这类「你项目里真实的 Lua 场景」是强加分。
- 限流算法速答：固定窗口（`INCR` + `EXPIRE`，实现最简，窗口边界可双倍突发）、滑动窗口（ZSET 存时间戳 + `ZREMRANGEBYSCORE` 后计数，精确但费内存）、令牌桶（恒速补 token、允许可控突发，Lua 原子实现）、漏桶（恒速流出、削峰整形）。计数器实现的经典陷阱：**只在 `INCR` 返回 1 时才 `EXPIRE`**——每次请求都刷新 TTL 会让持续使用的合法用户计数永不过期、累到上限后被长期误封（真实线上 bug）。
- 降级：Redis 不可用时的策略要分数据类型——缓存类直接穿透到 DB，账本 / 锁类必须失败快返回（宁可拒绝服务不可数据错乱），并说明这是显式配置的降级开关。

### 13.3 消息与异步任务（按项目情况选答）

**高频问法**：「文档入库这种慢任务怎么异步化？」「任务失败怎么重试？怎么防止重复消费？」「Redis Stream / RabbitMQ / Kafka 怎么选？」

**答题要点**：受理返回任务 id + 后台消费 + 状态机可查询；消费幂等（任务 id 去重）+ 重试上限 + 死信记录 + 补偿定时任务兜底；选型按规模——单机量级 Redis Stream / 后台任务足矣，跨系统解耦或高吞吐才上 MQ，体现「不过度设计」。

### 14.1 Vue3 核心

**高频问法**
- 「Composition API 相比 Options API 解决了什么？setup 语法糖下组件怎么组织？」
- 「ref 和 reactive 怎么选？解构 reactive 为什么丢响应性？」
- 「computed 和 watch / watchEffect 的区别与适用场景？」
- 「Vue3 响应式原理（Proxy）和 Vue2（defineProperty）的区别？」

### 14.2 SSE 流式消费与聊天 UI（AI 岗特色前端题）

**高频问法**
- 「前端怎么消费 SSE？EventSource 有什么局限，为什么常用 fetch 流式读？」
- 「流式 Markdown 怎么渲染？半个代码块 / 不完整语法怎么处理？」
- 「打字机效果怎么做到不卡顿？每个 chunk 都触发渲染吗？」
- 「断线重连怎么做？重连后内容会重复吗？」
- 「LLM 输出直接 v-html 渲染有什么风险？」

**答题要点**
- EventSource 仅 GET / 无自定义头（带不了 Authorization），生产常用 fetch + ReadableStream 手动解析 `event:` / `data:` 帧，注意跨 chunk 的半行拼接（buffer 累积按 `\n\n` 切分）。
- 渲染性能：chunk 累积节流渲染（rAF / 定时批量），Markdown 增量解析容错未闭合语法；长对话虚拟滚动。
- 按事件类型分发：delta（增量文本）/ reference（引用卡片）/ error / done，终态驱动 UI 状态机（loading → streaming → done/failed），发送按钮防重复提交。
- 安全：LLM / 服务端内容必须过 DOMPurify 再进 v-html，防 XSS——AI 前端的必考安全题。
- 断连处理：显式关闭旧连接防泄漏（组件卸载 / 路由切换时 abort）、重连按消息 id 去重或拉快照重放。

### 14.3 请求层与工程化

**高频问法**：「Axios 封装做了什么？401 刷新 token 的并发问题怎么处理？」「加载 / 空态 / 失败 / 重试四态怎么统一？」「前后端接口契约怎么保证不漂移？」

**答题要点**：拦截器统一注入鉴权、超时、错误转 Toast（支持 silent 静默模式避免重复弹窗）；401 刷新用单飞（一个刷新请求，其余挂起排队重放），刷新失败统一登出；契约核对路径 / 方法 / 分页 / 错误码 / 权限角色，联调用 OpenAPI 文档对齐。

## 十五、部署、DevOps 与可观测

> 考察定位：验证「你的项目真的上过线吗」。围绕 Docker、环境一致性、健康检查、日志监控、发布回滚提问。

**高频问法**
- 「Dockerfile 多阶段构建是什么？镜像怎么做小？」
- 「docker-compose 里服务启动顺序怎么保证？depends_on 够吗？」
- 「healthcheck 怎么写？没有它会发生什么？」
- 「前端 Nginx 怎么配 SPA 路由、API 代理和 SSE？」
- 「环境变量与密钥怎么管理？.env 会提交到 git 吗？」
- 「CI 流水线有哪些阶段？测试怎么分层跑？」
- 「上线出问题怎么回滚？数据库迁移能回滚吗？」
- 「日志怎么打才有用？trace_id 怎么贯穿全链路？」
- 「Token 成本监控怎么做？预算超了系统行为是什么？」

**答题要点**
- 多阶段构建：构建期依赖不进运行镜像（前端 build → nginx 静态托管；Python 装依赖层缓存优化）；镜像瘦身（slim 基底、.dockerignore、层合并）。
- depends_on 只管启动顺序不管就绪：`condition: service_healthy` + 各服务 healthcheck（pg_isready / redis-cli ping / HTTP /health）；应用自身健康检查要区分存活（liveness）与就绪（readiness，含依赖检查），健康端点豁免限流。
- Nginx：SPA fallback（try_files → index.html）、API 反代、**SSE 必须 `proxy_buffering off` + 长读超时**——SSE 上线后「本地流式、线上一次性吐」的经典事故。
- 配置纪律：.env 不进 git，提供脱敏 .env.example；生产镜像版本固定禁 latest；密钥走 Secret 管理。
- CI 分层：lint + 类型检查 → 单测（mock LLM）→ 集成测试（真实 PG / Redis 容器）→ 构建镜像 → 评测门禁（夜间全量）；用 markers 区分，live 测试不进默认流水线。
- 可观测三件套：结构化日志（JSON + trace_id 贯穿请求 → LLM 调用 → 工具执行）、指标（QPS / 延迟分位 / Token 成本 / 缓存命中率 / 预算水位）、告警（错误率、成本突增、供应商可用性）。
- 成本治理：逐请求记录 input/output token 与单价 → 按模块 / 租户归因 → 日预算护栏（原子预留 + 结算，超限拒绝并友好提示）；离线批量任务（评测夜跑 / 批量入库）错峰调度吃谷时价（见 2.7 峰谷定价）。

## 十六、项目深挖与追问链实战

> 考察定位：社招 60% 的时间在这里。面试官从简历上的项目出发连环下钻，验证真实性与深度。以下以「企业智能办公 Agent 平台（对话 + RAG + Agent + 工作流 + ERP 预审 + Prompt 调试 + 用量看板）」类项目为例，给出真实追问链与应答策略。

### 16.1 开场三连（必然发生）

1. 「**一分钟介绍这个项目**：背景、你的角色、核心技术、量化结果。」——准备 STAR 稿（见附录 C），结果必须量化（如：混合检索 + Rerank 将评测集 Recall@4 从 0.72 提到 0.89；精确缓存命中 35% 降低成本 28%——用你自己的真实数据，且口径必须与实现一致：项目做的是精确匹配缓存就不要说成语义缓存，被追问「相似度阈值怎么定的」会当场露馅）。
2. 「**架构图画一下**，讲讲每个模块为什么这么拆。」——分层画（前端 / 网关 / 服务 / LLM 编排 / 存储），每层讲一个关键决策。
3. 「**项目里你最有挑战的问题**是什么？怎么解决的？」——选有排查过程与量化对比的（如 SSE 生产环境缓冲问题、BM25 多 worker 索引失效、预算并发超卖）。

### 16.2 典型追问链示例（背结构不背答案）

**追问链 A：RAG 质量**
「知识库怎么做的」→「切片参数怎么定的」→「为什么 pgvector 不是 Milvus」→「混合检索为什么要加 BM25」→「RRF 的 k 怎么调」→「Rerank 加了多少延迟，值得吗」→「怎么证明这些优化有效」→「无答案的问题你们怎么处理」。
应对：每一环都有「决策依据 + 数据对比」，最后一环落到评测集与拒答策略。

**追问链 B：Agent 可控性**
「Agent 怎么实现的」→「LangGraph 的图长什么样」→「工具选错 / 参数幻觉怎么办」→「最多几步，超了呢」→「怎么算一次任务的成本」→「用户中途关页面任务怎么办」→「执行记录能回放吗」。
应对：护栏三件套（步数 / 预算 / 超时）+ 状态外置恢复 + trace 回放。

**追问链 C：并发与稳定性**
「QPS 多少」→「LLM 调用慢，并发上来会怎样」→「限流在哪层做的」→「Redis 挂了会怎样」→「预算控制并发下怎么不超卖」→「压测做过吗，瓶颈在哪」。
应对：诚实报真实规模，但方案要有并发正确性细节（Lua 原子预留、无状态水平扩、降级开关分级）。

**追问链 D：工程质量**
「测试怎么做的」→「LLM 相关代码怎么测」→「集成测试用真库还是 mock」→「上线流程与回滚」→「出过什么线上事故，怎么复盘的」。
应对：分层测试策略 + 一个真实事故复盘（现象 → 根因 → 修复 → 防复发机制）。

### 16.3 高危陷阱题

- 「这个项目是你一个人做的还是团队做的？你具体负责哪部分？」——诚实划边界，吹全栈会被逐模块验证。

- 「你觉得大模型应用开发未来 1-2 年的趋势？」——可答：Agent 从演示走向治理（评测 / 权限 / 成本）、MCP 生态标准化、推理模型改变 Prompt 范式、多模态与端侧、合规常态化——重点是每条能接一句自己的判断。
- 「业务方说 AI 答错了一个重要问题，老板要你保证不再出错，怎么回应？」——考预期管理：概率系统无法承诺零错误，但可以承诺**该类错误的闭环**（入评测集 + 针对性修复 + 灰度验证 + 监控告警），并给出准确率与兜底策略（人工复核通道）。
- 「排期很紧，PM 要求直接上线没有评测的功能，你怎么办？」——考工程底线与沟通：给出最小可行质量方案（冒烟评测集 + 灰度 + 回滚预案），量化风险给决策者选择，不是硬顶也不是裸奔。
- 「和算法 / 产品 / 后端同事对 AI 效果预期有分歧怎么处理？」——用数据说话：建立共同认可的评测标准，把「感觉不好」转化为可归因的 bad case。

## 附录 C：答题模板与 STAR 项目稿框架

**技术题四段式**：概念一句话 → 真实系统中的问题 → 生产方案与取舍 → 适用边界。

**场景排查题**：先定位段（归因）→ 按链路逐段给检查点 → 每个检查点给修复手段 → 修复后如何固化（评测 / 监控）防复发。

**系统设计题**：澄清需求（规模 / 延迟 / 合规 / 成本）→ 分层架构图 → 每层 1~2 个关键决策与备选对比 → 质量闭环（评测 / 观测 / 灰度回滚）→ 演进路线（当前规模的简单方案 → 10 倍规模的升级路径）。

**STAR 项目稿（3 分钟版）**
- S（背景，20s）：业务痛点与规模（谁在用、什么场景、原来多痛）。
- T（任务，10s）：你的目标与角色边界。
- A（行动，90s）：3 个关键技术决策，每个 = 问题 → 备选方案 → 为什么这么选 → 怎么实现。
- R（结果，30s）：量化指标（效果 / 性能 / 成本）+ 评测方法佐证。
- 反思（30s）：一个踩坑与防复发机制，一个如果重来会改的设计。

> 结语：2026 年的 AI 应用面试，本质是在验证一件事——**你能否对一个概率系统做确定性的工程治理**。检索不准能归因、Agent 失控有护栏、效果好坏有评测、成本延迟有账算、安全合规有底线。把这五句话对应的章节吃透，再用自己项目的真实数据填进去，就是最强的面试准备。
